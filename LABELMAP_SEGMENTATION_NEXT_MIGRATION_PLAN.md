## Goal & end state

When complete, an OHIF app running with `appConfig.useNextViewports: true` will render, **in place** on native `PLANAR_NEXT` viewports, without ever recreating the viewport as `ViewportType.ORTHOGRAPHIC`:

- **DICOM SEG / manual labelmap** — single non-overlapping labelmaps via the `legacy-stack-image` render plan; **overlapping** SEGs via `useSliceRendering` (multiple labelmap layers under one `segmentationId`, one depth-offset `vtkImageSlice` actor per layer, all overlapping segments simultaneously visible).
- **Contour** — already native via the cs3d `getViewportICamera` bridge (commit `06605d040`).
- **RTSS** — contour geometries render natively, AND survive the hydrate re-mount (the v1 plan wrongly marked this fully done; see **E3**).

The `SegmentationService` is refactored so all native-vs-legacy logic is delegated to a **backend twin** mirroring the existing viewport backend twins at `extensions/cornerstone/src/services/ViewportService/backends/` (`IViewportBackend` + `Legacy`/`NextViewportBackend`; `IViewportOperations` + `Legacy`/`Next` + `viewportOperations` dispatcher). The new twin is `ISegmentationBackend` (+ narrow `ISegmentationServiceInternals`) with `LegacySegmentationBackend` / `NextSegmentationBackend`.

**Flag-OFF is byte-identical.** The `LegacySegmentationBackend` holds the current decision tree moved verbatim (including the `ORTHOGRAPHIC` promotion); no new config field is set; the existing ~38 legacy e2e specs are unaffected. Read-only display (and overlap) is the primary deliverable; native brush/erase editing is an explicit, deferred follow-on.

## The 4 live errors — root causes

All four were re-traced at HEAD. **E1 and E2 share one root site** (`convertStackToVolumeViewport` at `SegmentationService.ts:1731`); **E3 is a different OHIF-side defect** (native mount drops `segmentationPresentation`); **E4 is the overlap-rendering gap** (`.flat()` collapse + no `useSliceRendering`). The single most important fact (**CONFIRMED in code during implementation 2026-06-20**): with `appConfig.useNextViewports` on and `appConfig.useGenericViewport` off, OHIF instantiates the **raw native `PlanarViewport`** — `getCornerstoneViewportType` maps `stack`→`PLANAR_NEXT` *directly* (`getCornerstoneViewportType.ts:41-46`), so the recorded `requestedType` is `PLANAR_NEXT` (not `STACK`), and `init.tsx:88` ties the adapter-selecting `rendering.useGenericViewport` to the **separate** `useGenericViewport` flag (off here). The raw `PlanarViewport` genuinely lacks `getViewPresentation`, so the throw is real. The `PlanarViewportLegacyAdapter` (which *defines* `getViewPresentation`) is selected only when a viewport is requested as `STACK`/`ORTHOGRAPHIC` with cs3D's compat remap on — which the next path never does, and which the user explicitly does not want. (This corrects the workflow's E1 finding, which assumed the adapter; v1's ledger verdict was right in OUTCOME — a throw — but for the wrong reason.)

### E1 — `addSegmentationRepresentation` reaches `getViewPresentation` / silently promotes a native stack viewport

- **Symptom:** `TypeError: viewport.getViewPresentation is not a function` when a SEG/manual labelmap is added to a native viewport — a genuine throw, because OHIF's next path runs on the raw `PlanarViewport` (see the confirmed fact above). (If the viewport were ever the legacy adapter instead, the same call site would silently recreate it as `ViewportType.ORTHOGRAPHIC` — also a defeat of `useNextViewports`. **CH3** prevents both.)
- **Root-cause site:** `SegmentationService.ts:1731` `const prevViewPresentation = viewport.getViewPresentation();` inside `convertStackToVolumeViewport` (`:1726`), which then `setViewPresentation` (`:1738`) and recreates the viewport via `viewportGridService.setDisplaySetsForViewport({ ...viewportType: ViewportType.ORTHOGRAPHIC })` (`:1753-1758`).
- **Call chain (verified):**
  1. `addSegmentationRepresentation` (`:286`) → for LABELMAP only, gated at `:322` → `determineViewportAndSegmentationType` (`:323` / `:1597`).
  2. `determineViewportAndSegmentationType:1598` `isVolumeViewportType(csViewport)` returns **false** for a native stack viewport, because `getLegacyViewportType.ts:30` returns `requestedType ?? type` = `STACK` and `isVolumeViewportType` is true only for `ORTHOGRAPHIC`/`VOLUME_3D` (`getLegacyViewportType.ts:58-66`).
  3. `handleViewportConversion:1615` `handler = isVolumeViewport ? handleVolumeViewportCase : handleStackViewportCase` → **`handleStackViewportCase`** (`:1644`).
  4. `handleStackViewportCase`: `isVolumeSegmentation` branch → `convertStackToVolumeViewport` (`:1655`); else try `updateLabelmapSegmentationImageReferences` (`:1659`, native-safe duck-typed resolver); else `attemptStackToVolumeConversion` (`:1663`) → on FrameOfReference match → `convertStackToVolumeViewport` (`:1780`).
  5. `convertStackToVolumeViewport:1731` → the defeater.
- **Why it throws (confirmed in code 2026-06-20):** OHIF's `getCornerstoneViewportType` maps `stack`/`volume`/`orthographic`→`PLANAR_NEXT` when `useNextViewports` is on (`getCornerstoneViewportType.ts:41-46`), so OHIF passes `PLANAR_NEXT` to the engine and the recorded `requestedType` is `PLANAR_NEXT` (not `STACK`). The class resolver returns `PlanarViewportLegacyAdapter` only for `PLANAR_NEXT` + `requestedType STACK|ORTHOGRAPHIC` (`viewportTypeToViewportClass.ts:123-132`); with `requestedType=PLANAR_NEXT` it falls through to the **raw `PlanarViewport`** (`grep -c getViewPresentation PlanarViewport.ts = 0`), so `convertStackToVolumeViewport:1731` throws for real. The adapter is reachable only via cs3D's compat remap (`rendering.useGenericViewport`, wired to the *separate* `appConfig.useGenericViewport` at `init.tsx:88`, **off** in the next config). So no dist rebuild or constructor probe is needed — the raw viewport is the intended next-path runtime class, and the user has asked to keep it that way (no adapter).
- **Critical trap for the fix:** `csUtils.isGenericViewport` returns **true** for BOTH the raw `PlanarViewport` AND the `PlanarViewportLegacyAdapter` (the adapter inherits `setDisplaySets`/`setDisplaySetPresentation`/`setViewState`; `viewportCapabilities.ts:118-126`), and **false** for legacy `StackViewport`/`VolumeViewport`. So `isGenericViewport` is the exact routing predicate. The backend must route on `isGenericViewport` and **never** on `try/catch` or method-presence around `getViewPresentation` — the adapter is a hybrid that *also* answers `getViewPresentation`, so a capability probe would misroute it back to legacy and re-introduce the flip.
- **Fixed by:** **CH3** (backend keystone) — `NextSegmentationBackend.classifyAndPrepareLabelmapAdd` returns `{ LABELMAP, isConverted: false }` and never calls `convertStackToVolumeViewport`.

## Architecture: the SegmentationService backend twin

### Why a twin, and which existing pattern it mirrors

The viewport service deliberately has **two** twin families (both re-verified at HEAD), and the seg twin merges their best traits:

- **`IViewportBackend`** (lifecycle/mount) — one interface, two **class** impls each injected with the host service via `constructor(private readonly service: IViewportServiceInternals)` (`LegacyViewportBackend.ts:33-34`, `NextViewportBackend.ts:75`). Dispatch is **flag-selected once**, lazily, via a memoized `private get backend()` getter (`CornerstoneViewportService.ts:153,169-176`). `IViewportServiceInternals` (`:17-43`) is a **narrow** 4-method interface the host `implements` and passes `this` into.
- **`IViewportOperations`** (per-viewport interaction) — one interface, two **stateless plain-object** consts (`legacyViewportOperations`/`nextViewportOperations`), dispatched **per-viewport** at call time via `viewportOperations.ts:20-22` `backendFor(vp) = csUtils.isGenericViewport(vp) ? next : legacy`.

The seg forks are **stateful and host-dependent** (they read `servicesManager`/`viewportGridService`/`displaySetService`), so they take `IViewportBackend`'s **class+injection shape**, not the stateless plain-object shape. But the seg viewport-bearing methods always have an already-resolved, self-describing `csViewport` in hand and a flag-ON session can **mix** lanes (overlays attach to whichever viewport is live), so dispatch is **per-viewport** via `csUtils.isGenericViewport` (like `viewportOperations`), **not** flag-selected. This is the one deliberate divergence from `IViewportBackend` and must be documented in the interface file exactly as `viewportOperations.ts` documents its own.

### Dispatch mechanism

Per-method dispatch, justified by whether a target viewport exists:

- **Per-viewport** via `csUtils.isGenericViewport(viewport)` for the three viewport-bearing methods (`classifyAndPrepareLabelmapAdd`, `convertSegmentationDataToVolumeIfNeeded`, `jumpToSegmentCenter`). `isGenericViewport` is true for both native flavors (raw `PlanarViewport` AND the adapter) and false for legacy `StackViewport`/`VolumeViewport`, so it is the exact predicate that routes every flag-ON stack/MPR viewport to the Next twin.
- **Session-flag** via `isNextViewportsEnabled()` for the one viewport-less method (`assembleSegmentationDataForSEG`, which runs at SEG-load before any target viewport exists).

**Construction (eager, simpler than the viewport backend's lazy getter):** because per-viewport dispatch has no per-session flag to defer on, `SegmentationService` constructs **both** twins once in its constructor — `this._legacySegBackend = new LegacySegmentationBackend(this); this._nextSegBackend = new NextSegmentationBackend(this);` — passing `this` (typed `ISegmentationServiceInternals`). A private getter `_segBackend(vp) = csUtils.isGenericViewport(vp) ? this._nextSegBackend : this._legacySegBackend` returns the lane per viewport. Eager is safe (unlike `CornerstoneViewportService:169-176`'s lazy flag getter) because the twins read state at call time (post-init), not at construction. Backends import the host as `import type { ISegmentationServiceInternals }` only and inline any constants (mirror `NextViewportBackend.ts:17-19`) to avoid the backend→service circular dependency.

### The viewport-creation boundary

`convertStackToVolumeViewport` **is** viewport (re)creation — it tears down the `PLANAR_NEXT` viewport and rebuilds it as `ViewportType.ORTHOGRAPHIC` by calling `viewportGridService.setDisplaySetsForViewport({...viewportType: ORTHOGRAPHIC})` (`:1753-1758`). **Decision:** the seg backend owns ONLY the labelmap-add **decision** (promote-to-volume vs render-in-place) and **delegates** viewport creation to `CornerstoneViewportService`. The `NextSegmentationBackend` never calls `convertStackToVolumeViewport` and never reimplements creation; the verified native rule is that **no viewport conversion is needed** — `resolveLabelmapRenderPlan` picks `legacy-stack-image` for a `PLANAR_NEXT` stack labelmap (no `volumeId`, no `VTK_VOLUME_SLICE` precondition) and `updateLabelmapImageReferences` (`:1659`, duck-typed) resolves the image mapping in place. The `LegacySegmentationBackend` keeps `convertStackToVolumeViewport` **verbatim**; it remains a private `SegmentationService` method the legacy twin reaches via `ISegmentationServiceInternals` (it touches `servicesManager`-owned services), so flag-OFF promotion is byte-identical.

**Clean ownership split, one touch-point:** (1) `ISegmentationBackend` owns the labelmap-add decision, SEG data assembly, the segmentation-DATA volume conversion (`convertStackToVolumeLabelmap`, pure data), jump-to-segment, and the contour/RTSS pass-through. (2) `IViewportBackend` (`CornerstoneViewportService`) keeps owning viewport creation / mount / presentation-restore. The two compose **only** through the public `addSegmentationRepresentation` API (the E3 replay calls it), never through each other's private internals.

## How native segmentation rendering works (verified)

**One render-plan dispatcher, four plan kinds.** All labelmap rendering funnels through `resolveLabelmapRenderPlan` (`packages/tools/src/tools/displayTools/Labelmap/labelmapRenderPlan/resolveLabelmapRenderPlan.ts:28-78`), which computes `useSliceRendering = shouldUseSliceRendering(segmentation, config)` (`:28`), `renderMode = getViewportLabelmapRenderMode(viewport,{useSliceRendering})` (`:29`), `isVolumeImageMapper = useSliceRendering && canRenderVolumeViewportLabelmapAsImage(viewport)` (`:32`), then selects one of `'unsupported'` (`:35`), `'volume-slice-image-mapper'` (`:49`), `'legacy-volume'` (`renderMode==='volume'`, `:60`), else `'legacy-stack-image'` (`:72`). Kinds enumerated in `types.ts:6-10`. **Two** call sites: `labelmapDisplay.ts:145` (→`reconcile`) and `addLabelmapToElement.ts:34` (→`mount`). OHIF only drives this seam — it reads only `segmentation`/`config`/duck-typed viewport methods, **no native/legacy viewport-type fork**.

**PLANAR_NEXT is first-class.** `getViewportLabelmapRenderMode.ts:93-101` (under `stateManagement/segmentation/helpers/`, **not** the render-plan dir — v1 path drift): for `PLANAR_NEXT`, return `'volume'` if `getVolumeId()` set, else `'image'` if `getCurrentImageId` is a function, else `'unsupported'`. **PLANAR_NEXT native overlay path (`legacy-volume` route).** `legacyVolumePlan.ts:174` short-circuits to `addLabelmapToPlanarGenericViewport` for a `PLANAR_NEXT` volume viewport *before* the legacy `addVolumesToViewports` call. `addLabelmapToPlanarGenericViewport` mounts each layer via `viewport.addDisplaySet(dataId,{orientation,role:'overlay'})` (`planarGenericVolumeLabelmap.ts:128`) + `setDisplaySetPresentation` (`:132`), using `getViewReference`/`setViewReference`, never `getCamera`. **Precondition v1 missed:** this path early-returns (mounts nothing) unless the SOURCE actor renderMode is `CPU_VOLUME` or `VTK_VOLUME_SLICE` (`getPlanarNextVolumeRenderMode`, `:78-80,161-166`). ## Open questions / risks

1. **Does `updateLabelmapSegmentationImageReferences:1659` resolve in place on a native `PLANAR_NEXT` viewport at the add/overlay-mount instant?** Duck-typed (`getCurrentImageId`/`isReferenceViewable` both on `PlanarViewport`), so it *should*, but the FoR/SOP-match behavior — and whether `getCurrentImageId` is stale immediately after the E2 `setDisplaySets` swap — is not determinable by reading code. CH3 returns `isConverted:false` regardless, so it never promotes; but whether the SEG renders in that instant (vs late/empty) is the decisive CH0/CH4 runtime observation.
2. **VTK_VOLUME_SLICE / CPU_VOLUME guarantee for overlap (CH8).** Two silent-failure preconditions (slice-path `getPlanarPrimaryRenderMode===VTK_VOLUME_SLICE`; native-overlay-path source actor renderMode). If OHIF's SEG viewports don't reliably report them, overlap degrades to single-id invisibly — the single biggest silent-failure risk for the overlap deliverable.
3. **Segment→layer binding survival (CH7).** The adapter *produces* per-group segment data (`labelmapImagesFromBuffer.ts:690-697`) but its survival onto `segDisplaySet` after `Object.assign(results)` (`getSopClassHandlerModule.ts:163`) is unverified. Must confirm before building the overlap branch; the non-overlap path needs none of it, and E1/E2/E3 are independent.
4. **Skip vs `getViewState`/`setViewState` rewrite for `convertStackToVolumeViewport` on native (CH3).** Do any callers OUTSIDE `SegmentationService` (MPR/crosshairs) rely on the viewport becoming `ORTHOGRAPHIC` as a side effect of a labelmap add? If so, a plain native skip changes their behavior (flag-ON only); grep consumers of post-add viewport type before merge.
5. **Native MPR data-volume conversion policy (`convertSegmentationDataToVolumeIfNeeded`).** Default-to-skip keeps stack labelmaps so cs3d picks `legacy-stack-image`/native overlay. Revisit only if a real native-MPR SEG config needs the volume labelmap.
6. **E3 double-add / color-LUT preservation (CH6).** Confirm `cstSegmentation.addSegmentationRepresentations` is idempotent per `{viewportId,segmentationId,type}` if both `_setSegmentationPresentation` and (a future HP-retained) `_addOverlayRepresentations` fire; confirm the re-added native contour keeps its per-ROI color LUT.
7. **Export round-trip ordering (CH11).** Until CH11 lands, exporting an overlapping native SEG silently mis-encodes (`labelmaps2D[z++]` collision). Gate/warn on multi-layer SEG export.
8. **`getOrCreateMergedStackLabelmapVolume` interaction (CH7).** If OHIF ever supplies a single layer carrying multiple labelmap images per referenced imageId, `labelmapLayerStore.ts:97-177` may silently fold them into one merged volume, collapsing overlap. Guard/assert when building the multi-layer map OHIF-side.
9. **PMAP-as-overlay latent ambiguity.** PMAP is safe today (volume display set, not `isOverlayDisplaySet`), but `commandsModule.ts:323` flags PMAP-as-SEG as unhandled; the modality-blind `Modality==='SEG' ? Labelmap : Contour` routing (`CornerstoneViewportService.ts:1601`) would send any future overlay modality through the contour seg path.
10. **Two-twin shape divergence must be documented.** Reviewers expecting `IViewportBackend`'s flag-selected single instance must be told why the seg twin uses eager dual instances + per-viewport dispatch (sessions mix lanes; a viewport is always in hand), and that E3 lives in the **viewport** backend, not the seg twin.
