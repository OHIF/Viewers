# CLI-era extension → 2026 plugin contract

Standalone extensions and modes generated by the old OHIF CLI
(`yarn cli create-extension` / `create-mode`) do not load under the 2026
plugin contract without changes. The mechanical work is done for you by

```bash
pnpm create ohif@beta migrate ./path-to-your-plugin --dry-run
```

`create-ohif migrate` reads your package, prints a **report** of every change
it will make (each line tagged `[rewrite]`, `[ok]`, or `[flag]`), and — when
you re-run it without `--dry-run` — applies the `[rewrite]` items. That report
**is** the migration checklist.

**This page is the why behind the report.** Each section below names a report
item, quotes the CLI-era template it came from, and explains what the 2026
contract requires and the failure it prevents. Read it when a report line is
not self-explanatory, or when a `[flag]` item asks you to change something the
tool will not touch automatically.

:::tip The fastest path is to not migrate at all
If your plugin is small, regenerate it with `pnpm create ohif@beta` (choose
_extension_ or _mode_) and move your `src/` over. You get the current template
verbatim and skip the archaeology. Everything below is the in-place
alternative for plugins too large to re-home by hand.
:::

## Why the old output never really worked

Before the item-by-item walkthrough, one finding motivates the whole rewrite.
The CLI-era build config declared the bundle's UMD library name **twice**
(`platform/cli/templates/extension/.webpack/webpack.prod.js`, lines 14–23):

```js title=".webpack/webpack.prod.js (CLI-era — BEFORE)"
  output: {
    library: {
      type: 'umd',
      umdNamedDefine: true,
    },
    path: outputFolder,
    filename: outputFile,
    library: pkg.name,
    chunkFilename: '[name].chunk.js',
    globalObject: "typeof self !== 'undefined' ? self : this",
  },
```

An object has one value per key, so the second `library: pkg.name` (a plain
string) silently overwrites the first `library: { type: 'umd', … }` object.
Webpack fell back to its default library type, so the CLI-era build **never
reliably emitted the UMD bundle its own comments documented.** The host loads
plugins as UMD globals; a build that does not dependably produce one cannot be
loaded at runtime. That is the root reason the build config is replaced
wholesale rather than patched.

## package.json

### `main` → `dist/index.umd.js`

**Before** (`platform/cli/templates/extension/dependencies.json`): no `main`
at all — only `"module": "src/index.tsx"`. The build wrote to
`dist/umd/<name>/`.

The contract build (`.rspack/rspack.prod.js`) emits a single
`dist/index.umd.js`, and `main` must point at it. The nested
`dist/umd/<name>/` path is gone. `module` still points at your source entry so
that directory-mode consumers (a `directory` entry in `pluginConfig.json`) can
compile your `src/` directly.

### `files`, `keywords`, `publishConfig`

**Before:** the CLI-era manifest had no `files`, no `publishConfig`, and only
the `keywords: ["ohif-extension"]` array.

The contract requires:

```json title="package.json (2026 contract — AFTER)"
{
  "keywords": ["ohif-extension"],
  "main": "dist/index.umd.js",
  "module": "src/index.tsx",
  "files": ["dist", "src", "public", "README.md"],
  "publishConfig": {
    "access": "public",
    "main": "dist/index.umd.js",
    "module": "dist/index.umd.js"
  },
  "peerDependencies": {
    "@ohif/core": "^3.13.0-beta.0",
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  }
}
```

This is the `create-ohif` extension template output verbatim (the template is
normative). Note what is **not** here:

- **No `publishConfig.types` and no `exports` map.** The plugin toolchain does
  not emit `.d.ts` files, so there is nothing to point a `types` field at, and
  the host resolves the UMD via `main`/`module`, not an `exports` map. Adding
  either would advertise a file no build step produces.
- `files` ships `src` as well as `dist` because directory-mode consumers
  compile from source.
- `keywords` carries `ohif-extension` (or `ohif-mode` for a mode) — registry
  discovery and `pnpm plugin doctor` key off it.

`publishConfig` rewrites only `main`/`module` at publish time so the published
tarball points consumers at the built UMD while the working tree keeps pointing
at source. **This rewrite only happens under `pnpm publish`** — the `npm` CLI
ignores `publishConfig` field overrides, so publish with pnpm.

### peerDependencies

**Before** (`dependencies.json`, lines 16–28): the CLI-era peer set wrongly
included the bundler itself —

```json title="dependencies.json (CLI-era — BEFORE)"
  "peerDependencies": {
    "@ohif/core": "^{LATEST_OHIF_VERSION}",
    "@ohif/extension-default": "^{LATEST_OHIF_VERSION}",
    "@ohif/extension-cornerstone": "^{LATEST_OHIF_VERSION}",
    "@ohif/i18n": "^1.0.0",
    "prop-types": "^15.6.2",
    "react": "^18.3.1",
    "react-dom": "^18.3.1",
    "react-i18next": "^12.2.2",
    "react-router": "^6.23.1",
    "react-router-dom": "^6.23.1",
    "webpack": "5.89.0",
    "webpack-merge": "^5.7.3"
  }
```

The contract:

- **Build tooling is never a peer.** `webpack` / `webpack-merge` are dropped;
  the `.rspack` toolchain lives in `devDependencies`. A consumer installing
  your plugin should not be asked to satisfy a bundler.
- **`@ohif/*` peers pin to the host range** (`^3.13.0-beta.0`, one convention
  for the whole plugin). The runtime loader refuses to load a plugin whose
  declared `@ohif/*` range does not admit the host version, so a stale range is
  a load-time failure, not a warning.
- **Drop the `@ohif/ui` peer** if present. `@ohif/ui` is legacy and the host
  does not hand it to runtime plugins; port UI to `@ohif/ui-next` (the APIs
  differ — this is a `[flag]`, migrate will not rewrite your imports).

### `.npmrc`

The migration adds an `.npmrc` with `auto-install-peers=false`. The peers above
are provided by the host at runtime; pnpm must not auto-install its own copies,
which would produce duplicate React/`@ohif/core` instances.

## Build config: `.webpack/` → `.rspack/`

The old `.webpack/webpack.prod.js` is removed and replaced with the current
template's `.rspack/rspack.prod.js` plus a vendored `.rspack/pluginExternals.js`.
Two parts of that config **are** the contract — the host loads your bundle
against them — and they are the reason a hand-patch is not enough.

### output.library

Instead of the doubled-key output above, the contract config declares the UMD
library once, correctly:

```js title=".rspack/rspack.prod.js (2026 contract — AFTER)"
  output: {
    path: path.resolve(__dirname, '../dist'),
    filename: 'index.umd.js',
    library: { name: pkg.name, type: 'umd', export: 'default' },
    globalObject: "typeof self !== 'undefined' ? self : this",
    clean: true,
  },
```

`export: 'default'` makes the UMD global (`window[pkg.name]`) the extension
object itself, with no `.default` indirection for the host to unwrap.

### externals — the breaking change

This is the single change most likely to break a CLI-era plugin at runtime.

**Before** (`webpack.prod.js`, lines 25–46): externals were hand-rolled
per-target objects covering only `react`, `@ohif/core`, and `@ohif/ui` —

```js title=".webpack/webpack.prod.js (CLI-era — BEFORE)"
  externals: [
    {
      react: {
        root: 'React',
        commonjs2: 'react',
        commonjs: 'react',
        amd: 'react',
      },
      '@ohif/core': {
        commonjs2: '@ohif/core',
        commonjs: '@ohif/core',
        amd: '@ohif/core',
        root: '@ohif/core',
      },
      '@ohif/ui': {
        commonjs2: '@ohif/ui',
        commonjs: '@ohif/ui',
        amd: '@ohif/ui',
        root: '@ohif/ui',
      },
    },
  ],
```

Everything not in that list — `@cornerstonejs/*`, `dcmjs`, `gl-matrix`,
`react-dom`, `react/jsx-runtime` — was **bundled into the plugin.** A plugin
that bundles its own React DOM or its own Cornerstone runs a second copy
alongside the host's, which breaks hooks, tool state, and rendering in ways
that surface only at runtime.

The contract externalizes the full host-provided surface:

```js title=".rspack/pluginExternals.js (2026 contract — AFTER)"
module.exports = [
  /\b(vtk.js)/,
  /\b(dcmjs)/,
  /\b(gl-matrix)/,
  /^@ohif/,
  /^@cornerstonejs/,
  'react',
  'react-dom',
  'react/jsx-runtime',
];
```

`react`, `react-dom`, and `react/jsx-runtime` are now host-provided globals.
If your source imported any of the newly-external packages expecting them
bundled, nothing changes in your code — but the runtime now supplies the
host's single copy, which is the point.

The vendored `pluginExternals.js` is a verbatim copy of the monorepo's
`.rspack/pluginExternals.js`; `create-ohif` keeps the two in sync. See
[Webpack to Rspack v2](./3p12-to-3p13/build-tooling.md) for the app-level side
of the same bundler move.

### babel / other build files

`babel.config.js` and the `@babel/*` dependencies are no longer used — the
`.rspack` build compiles TS/JSX with `builtin:swc-loader`. Any other custom
build file (`webpack.config.js`, a `vite.*`/`rollup.*` config, extra files in
`.webpack/`) is **flagged, not rewritten**: migrate leaves it in place and you
port any custom settings into `.rspack/rspack.prod.js` by hand.

## The extension object: id and default export

The extension must `export default` an object whose `id` equals the
`package.json` `name`. The CLI-era template already did both (`src/index.tsx`
default-exports an object with `id` sourced from `src/id.js`), so most ports
only need to **verify** it — the loader rejects a plugin whose `id` does not
match its package name. migrate does not rewrite your source object; this is a
verification step.

## Registration and discovery

The CLI commands that used to register and find plugins are gone. Their
replacements:

- `yarn cli add-extension <pkg>` → `pnpm plugin add <pkg>`. This writes an
  explicit entry into
  [`platform/app/pluginConfig.json`](../platform/extensions/pluginConfig.md).
  Delete any dead `version` field from an old hand-written entry — the config
  schema now rejects unknown fields. See
  [Extension installation](../platform/extensions/installation.md) for the
  from-npm, from-checkout, and in-tree forms.
- **Load without a rebuild** — declare the plugin as a runtime descriptor in
  `app-config.js` instead. See
  [Runtime Extensions](../platform/extensions/runtime-extensions.md); the load
  is recorded at `window.__ohif.runtimeExtensions` for auditing.
- `yarn cli search` → the [extension gallery] plus `npm search ohif-extension`
  (the registry keyword query the CLI search ran).

[extension gallery]: ../platform/extensions/index.md

## Styles

CLI-era extensions that relied on the host's Tailwind classes must now ship
self-contained CSS. The in-tree host content globs cover `pluginConfig`
entries, but published and runtime-loaded plugins carry their own CSS through
the build (the `.rspack` CSS pipeline emits `index.css`) and, for runtime
descriptors, the descriptor `styles: []` field. migrate adds a
`tailwind.config.js` (with preflight off) for extensions that lack one.

## Verify

After the rewrites:

```bash
pnpm install
pnpm build
pnpm plugin doctor   # peer-range and keyword checks
```

then load the plugin in the viewer and confirm it registers (for a
runtime-loaded plugin, check `window.__ohif.runtimeExtensions` for its audit
record). The `pnpm plugin` helpers replace the deleted CLI; see the
[OHIF CLI removal](./3p12-to-3p13/cli-removal.md) for the full command
mapping.
